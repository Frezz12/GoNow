# GoNow: рабочий процесс Git и AI-агентов

Этот файл — обязательное правило работы для всех участников GoNow: разработчиков и AI-агентов. Его цель — сохранять `main` стабильной, не терять изменения и делать каждую доработку проверяемой.

## Основной принцип

`main` — стабильная ветка. В неё не выполняются обычные прямые изменения. Каждая самостоятельная задача выполняется в отдельной ветке, проходит проверку и попадает в `main` через Pull Request.

```text
main → рабочая ветка → реализация → проверки → commit → push → Pull Request → review → merge в main
```

Одна ветка должна решать одну связанную задачу. Например, экран профиля и его API могут быть одной задачей; переработка авторизации и визуальный редизайн карты — разными.

## Имена веток

Используются короткие имена на английском, разделённые дефисами.

| Тип | Когда использовать | Пример |
| --- | --- | --- |
| `feat/` | Новая функция | `feat/profile-media` |
| `fix/` | Исправление ошибки | `fix/email-code-delivery` |
| `docs/` | Документация | `docs/api-profile-media` |
| `refactor/` | Изменение архитектуры без новой функции | `refactor/profile-feature-structure` |
| `chore/` | Конфигурация, CI, зависимости | `chore/ios-build-check` |
| `codex/` | Любая ветка, созданная AI-агентом | `codex/feat/profile-media` |

AI-агент всегда использует префикс `codex/`. Пример: `codex/fix/map-create-button`.

## Начало задачи

Перед работой необходимо убедиться, что рабочая директория чистая, обновить `main` и создать ветку от актуального `origin/main`.

```bash
git status -sb
git fetch origin
git switch main
git pull --ff-only origin main
git switch -c codex/feat/short-task-name
```

Если в рабочей директории есть чужие или непонятные изменения, их нельзя удалять, перезаписывать или включать в свой коммит без явного согласования.

## Правила для AI-агента

1. До начала изменений прочитать относящиеся к задаче код, документацию и конфигурацию.
2. Работать только в собственной ветке `codex/<type>/<task>`.
3. Не менять `main` напрямую и не выполнять force-push в `main`.
4. Не использовать `git reset --hard`, `git checkout --` и другие разрушительные команды без явного запроса владельца проекта.
5. Не коммитить `.env`, ключи Resend/S3, JWT-секреты, пароли, токены, локальные базы, DerivedData или личные файлы IDE.
6. При изменении API обновлять `API.md` в том же коммите. При изменении запуска или окружения обновлять `backend.md`, `.env.example`, `storage.md` или `README.md` по смыслу.
7. При изменении визуального языка обновлять `style.md`, если изменение влияет на общие правила интерфейса.
8. Перед коммитом запускать подходящие проверки и сообщать их результат.
9. Коммитить и отправлять ветку только по запросу владельца проекта или если это явно входит в поставленную задачу.
10. После push создавать Pull Request либо передавать владельцу ссылку на ветку, если создание PR не запрошено.

## Работа в одной задаче

Небольшие осмысленные коммиты проще проверять, откатывать и переносить. Сообщения пишутся в формате Conventional Commits:

```text
feat: add profile photo upload
fix: allow delete requests in CORS
docs: describe media API endpoints
refactor: separate profile media repository
```

Перед каждым коммитом:

```bash
git status --short
git diff --check
git add <только нужные файлы>
git diff --cached --check
git commit -m "feat: short description"
```

`git add -A` допустим только после проверки, что в рабочей директории нет посторонних файлов и секретов.

## Обновление рабочей ветки

Если `main` обновилась во время работы, сначала получить изменения:

```bash
git fetch origin
git rebase origin/main
```

Rebase применяется только к собственной рабочей ветке. Если эта ветка уже отправлена в GitHub, после rebase используется:

```bash
git push --force-with-lease
```

Обычный `git push --force` запрещён: он может удалить чужую историю. Если rebase вызывает конфликт, конфликт разрешается осознанно, затем повторно запускаются проверки. Если назначение конфликтующих изменений неясно, работа останавливается до согласования.

## Pull Request

Каждая ветка открывает PR в `main`. В PR необходимо указать:

- краткое описание результата;
- список ключевых изменений;
- как проверить функцию;
- выполненные команды проверки и их результат;
- миграции БД, новые переменные окружения и изменения API;
- ограничения, известные риски и следующий шаг, если работа неполная.

Для GoNow используется **squash merge**: одна завершённая задача становится одним чистым коммитом в `main`. После merge рабочая ветка удаляется.

## Настройки GitHub для `main`

В GitHub следует включить Branch Protection или Ruleset для `main`:

- Require a pull request before merging;
- Require 1 approving review;
- Require status checks before merging;
- Require conversation resolution;
- запрет прямых push, force-push и удаления ветки;
- разрешить squash merge и автоматическое удаление ветки после merge.

Минимальные проверки проекта:

```bash
# Backend
cd apps/backend
cargo fmt --check
cargo test

# iOS
cd ../..
xcodebuild -project apps/ios/GoNow/GoNow.xcodeproj \
  -scheme GoNow -sdk iphonesimulator build CODE_SIGNING_ALLOWED=NO
```

## Параллельная работа

Если backend, iOS и Android могут работать независимо, для них создаются разные ветки и PR:

```text
codex/feat/backend-profile-media
codex/feat/ios-profile-media
codex/feat/android-profile-media
```

Сначала согласуется контракт в `API.md`. Изменение API и backend-реализация объединяются раньше или одновременно с клиентскими PR. Миграции должны иметь уникальные имена и не редактироваться задним числом после попадания в `main`.

Не следует одновременно менять один и тот же большой файл без договорённости. Если пересечение неизбежно, изменения разбиваются по порядку: сначала общая модель/API, затем отдельные клиенты.

## Конфигурация и секреты

- Локальные секреты хранятся только в `.env` и не отправляются в Git.
- `.env.example` содержит только названия переменных и безопасные значения-заполнители.
- Новая переменная окружения документируется рядом с назначением в `backend.md` или `storage.md`.
- Access key, secret key, API key и JWT secret никогда не попадают в коммиты, PR, логи или скриншоты.

## Завершение задачи

После merge:

```bash
git switch main
git pull --ff-only origin main
git branch -d codex/feat/short-task-name
```

В `main` остаётся только проверенный код. Любая следующая задача начинается с новой ветки от актуального `main`.
