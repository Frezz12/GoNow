# GoNow Weather — инструкция для Android AI-агента

Этот документ описывает **готовую погодную интеграцию GoNow**. Реализуйте Android-клиент в соответствии с ним, не меняя контракт backend API без отдельного согласования.

## 1. Цель и архитектура

Погодный виджет на карте показывает погоду **в текущем месте пользователя**, а не в городе, указанном в профиле.

Поток данных:

```text
Android location provider
        ↓ latitude / longitude
GoNow backend: GET /api/v1/weather/current
        ↓
weather provider + reverse geocoding (только на сервере)
        ↓
город + погода
        ↓
Android weather widget
```

Ключевое правило: **Android-приложение не должно обращаться напрямую к Open-Meteo, Nominatim или другим погодным/геокодинговым сервисам.** Клиент обращается только к GoNow backend. Это уменьшает зависимость от VPN/DNS на телефоне и позволяет менять внешнего провайдера без обновления приложения.

## 2. Backend API

### Запрос

`GET {API_BASE_URL}/weather/current`

`API_BASE_URL` уже включает `/api/v1`. Для локальной разработки это:

```text
http://10.0.2.2:8080/api/v1
```

`10.0.2.2` — адрес хоста для Android Emulator. На физическом устройстве используйте доступный устройству HTTPS-адрес backend, а не `localhost`.

Параметры query:

| Параметр | Тип | Обязателен | Значения |
| --- | --- | --- | --- |
| `latitude` | `Double` | Да | от `-90` до `90` |
| `longitude` | `Double` | Да | от `-180` до `180` |
| `unit` | `String` | Да | `celsius` или `fahrenheit` |

Авторизация для этой ручки не требуется. Не добавляйте токен как обязательное условие получения погоды.

Пример:

```text
GET /api/v1/weather/current?latitude=55.7558&longitude=37.6173&unit=celsius
```

### Успешный ответ `200 OK`

```json
{
  "data": {
    "city": "Москва",
    "temperature": 21.4,
    "apparentTemperature": 20.9,
    "relativeHumidity": 58.0,
    "windSpeed": 8.4,
    "unit": "celsius",
    "weatherCode": 1,
    "isDay": true
  }
}
```

`city` может быть `null`, если погодный прогноз получен, но сервис определения города временно недоступен. В таком случае показывайте название города, определённое локально по координатам; если и оно недоступно — нейтральное «Текущее местоположение». Не подставляйте город из профиля к координатам телефона: это создаёт неверные пары вроде «Москва · 13°» при погоде Сан-Франциско.

### Ошибки

| HTTP-код | Значение | Поведение клиента |
| --- | --- | --- |
| `422` | Неверные координаты или единица | Не повторять автоматически; записать ошибку разработки. |
| `503` | Погодный провайдер временно недоступен | Показать «Погода недоступна», дать обновить позднее. |
| Нет сети / timeout | Backend недоступен | Показать «Нет подключения». |

Все успешные ответы обёрнуты в поле `data`, ошибки — в `error`. Сохраняйте `requestId` ошибки в диагностических логах.

## 3. Kotlin-модели

Пример для Retrofit + kotlinx.serialization (допустимы Moshi/Gson при сохранении JSON-полей):

```kotlin
@Serializable
data class ApiEnvelope<T>(val data: T)

@Serializable
data class WeatherDto(
    val city: String? = null,
    val temperature: Double,
    val apparentTemperature: Double,
    val relativeHumidity: Double,
    val windSpeed: Double,
    val unit: String,
    val weatherCode: Int,
    val isDay: Boolean
)

interface WeatherApi {
    @GET("weather/current")
    suspend fun getCurrentWeather(
        @Query("latitude") latitude: Double,
        @Query("longitude") longitude: Double,
        @Query("unit") unit: String
    ): ApiEnvelope<WeatherDto>
}
```

Не округляйте координаты до отправки. Температуру и другие значения округляйте только при отображении.

## 4. Геолокация на Android

### Разрешения

Добавьте в `AndroidManifest.xml`:

```xml
<uses-permission android:name="android.permission.ACCESS_COARSE_LOCATION" />
<uses-permission android:name="android.permission.ACCESS_FINE_LOCATION" />
```

После успешного входа пользователь попадает на вкладку карты. При первом показе карты:

1. Если разрешения нет — запросите `ACCESS_FINE_LOCATION` вместе с `ACCESS_COARSE_LOCATION` через runtime permission.
2. Если разрешение выдано — получите текущую точку через `FusedLocationProviderClient`.
3. Отправьте широту и долготу в GoNow API.
4. Полученный `city` используйте как главный источник названия города в погодном виджете.

Не запрашивайте фоновую геолокацию. Виджет обновляется, пока карта видна пользователю.

### Обновления при перемещении

Пользователь может приехать из Москвы в другой город или страну. Поэтому во время показа вкладки карты подпишитесь на обновления местоположения:

- приоритет: `Priority.PRIORITY_BALANCED_POWER_ACCURACY`;
- минимальное смещение: примерно `1_000` метров;
- остановить обновления, когда карта скрыта или экран уничтожен;
- при новой точке снова запрашивать `/weather/current`.

Не делайте запросы на каждый GPS-апдейт. Дополнительно ограничьте вызов API: не обновляйте погоду, если новая точка отличается менее чем на `0.001` градуса от последней точки и прошлый успешный ответ моложе 10 минут.

### Запасные сценарии

| Состояние | Что показывать |
| --- | --- |
| Разрешение ещё запрашивается | «Определяем место» + индикатор загрузки. |
| Пользователь запретил доступ, но в профиле есть сохранённые координаты | Погоду для точки профиля. |
| Пользователь запретил доступ и сохранённой точки нет | «Разрешите геолокацию». |
| Точка получена, город ещё определяется | «Определяем город…». |
| Backend вернул `city` | Использовать его всегда: он соответствует погодным координатам. |

Автоматически полученные текущие координаты **не нужно записывать в профиль**. Профильная точка — отдельная пользовательская настройка и применяется только как fallback.

## 5. Единицы температуры

В настройках приложения хранится один параметр:

```kotlin
enum class TemperatureUnit { AUTOMATIC, CELSIUS, FAHRENHEIT }
```

- `CELSIUS` → в API передавать `celsius`;
- `FAHRENHEIT` → в API передавать `fahrenheit`;
- `AUTOMATIC` → выбирать `fahrenheit` для US measurement system, иначе `celsius`.

Показывайте температуру как целое число с `°`, например `21°`. В подробностях добавляйте `°C` или `°F`.

## 6. Интерфейс виджета на карте

Стиль GoNow — нативный **Liquid Glass**. В Android воспроизводите его средствами Material 3/Compose: полупрозрачная поверхность, мягкое размытие доступное на версии ОС, светлая граница, аккуратная тень и крупные скругления. Не используйте непрозрачную прямоугольную карточку.

### Свёрнутое состояние

В левом верхнем углу карты разместите компактную стеклянную плашку:

```text
[иконка погоды] 21°
```

- Без текста «Солнечно» и без стрелки раскрытия.
- Вся плашка — кнопка размером не менее 44dp по высоте.
- Нажатие на любую часть плашки плавно раскрывает подробности.

### Раскрытое состояние

Плашка расширяется из своей позиции как стеклянная поверхность, без отдельного диалога:

```text
[иконка] Москва                         21°C
          Солнечно
────────────────────────────────────────────
  [термометр] 20°C       [влажность] 58%       [ветер] 8 км/ч
  Данные места © OpenStreetMap contributors
```

- Город — `WeatherDto.city`.
- Состояние погоды — по `weatherCode`.
- В подробностях: ощущаемая температура, влажность и скорость ветра.
- Повторное нажатие сворачивает плашку.
- Анимация 200–300 мс; при системном reduced motion — без сложной анимации.
- Укажите атрибуцию OpenStreetMap только в подробном состоянии, как в примере выше.

### WMO-коды и иконки

Используйте один набор Material Symbols / векторных иконок:

| WMO-коды | Состояние | Рекомендуемая иконка |
| --- | --- | --- |
| `0` | ясно | `wb_sunny` днём, ночная иконка ночью |
| `1–2` | переменная облачность | `partly_cloudy_day` / ночной вариант |
| `3` | пасмурно | `cloud` |
| `45–48` | туман | `foggy` |
| `51–57` | морось | `grain` или `rainy_light` |
| `61–67`, `80–82` | дождь | `rainy` |
| `71–77`, `85–86` | снег | `weather_snowy` |
| остальные | гроза | `thunderstorm` |

Не используйте emoji как иконки.

## 7. Поиск на карте: соседний UI-режим

Рядом с погодой находится профиль. При нажатии нижней круглой кнопки поиска погода и профиль скрываются, а вместо них появляется одна стеклянная капсула поиска. Это **только UI-заглушка**, запросы поиска пока не реализованы.

```text
[лупа]  Поиск задач и активностей                         [×]
```

- Поле сразу получает фокус.
- Кнопка `×` закрывает режим и возвращает погоду и профиль.
- Поисковая строка не заменяет нижнюю кнопку «Создать».

## 8. Тестирование

Проверьте минимум следующие сценарии:

1. Разрешить геолокацию и передать в Android Emulator Москву: `55.7558, 37.6173` — виджет должен показать «Москва» и московскую погоду.
2. Сменить точку эмулятора на Сан‑Франциско: `37.7749, -122.4194` — город и погода должны обновиться вместе.
3. Запретить геолокацию с сохранённой профильной точкой — виджет использует её.
4. Запретить геолокацию без профильной точки — виджет просит выдать разрешение.
5. Выключить интернет — отображается понятное состояние «Нет подключения».
6. Открыть и закрыть подробности погоды; проверить light/dark mode, крупный шрифт и reduced motion.
7. Открыть и закрыть поиск на карте; погодный виджет и профиль должны корректно возвращаться.

## 9. Что нельзя менять без согласования

- путь `/api/v1/weather/current` и названия JSON-полей;
- обёртку ответа `data`;
- правило «только GoNow backend обращается к внешним погодным сервисам»;
- отсутствие сохранения live-геолокации в профиль без явного действия пользователя;
- публичность weather endpoint: не делайте наличие access token обязательным на клиенте.

При расширении API одновременно обновляйте [API.md](API.md) и этот файл.
